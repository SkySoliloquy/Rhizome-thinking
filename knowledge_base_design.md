# 个人知识库系统设计文档

> 版本 v1.1 | 设计阶段，暂不涉及实现细节

---

## 一、设计目标

本系统服务于一个核心需求：**从自己过去的思想中找到新的连接点，以支持持续的深度思考**。

它不是归档系统，不是笔记本，不是发布平台。它是一个思维的外部扩展，允许用户在任意时刻捕捉输入，并在未来以任意角度重新发现这些输入与新问题之间的关系。

### 使用者画像

- 阅读范围跨领域（AI 研究论文、哲学、科幻小说、文章等）
- 思考碎片频繁但分散，想法有时明确，有时模糊
- 查询频率高（每天或每两天），目的是为新问题寻找既有思考的支撑或矛盾
- 不希望在维护系统本身上消耗注意力

### 访问方式要求

- **手机**：优先，使用频率最高，需要接近原生应用的体验
- **电脑**：支持，网页或应用均可，优先级低于手机

### 服务器环境

系统部署在专用小主机上，7×24 小时运行：

| 项目 | 规格 |
|------|------|
| CPU | Intel i5-8400 |
| 内存 | 4GB RAM |
| 存储 | 512GB |
| 操作系统 | Linux |

内存是主要约束。LLM 推理走云端 API（MiniMax），不占本地内存，本地服务合计预估占用 750MB - 1GB，在 4GB 限制内可行。

### 明确排除的需求

- 考研相关知识整理（由其他系统承担）
- 公开发布内容（无此需求）
- 语音输入（瓶颈不在输入速度）

## 二、核心数据模型：节点（Node）

系统中所有信息的基本单元是**节点**。每一条输入，无论来自阅读、随想还是构想，最终都被处理并存储为一个节点。

### 节点结构

```
Node
├── id                   全局唯一标识符（UUID）
├── timestamp            创建时间
├── source               来源信息
│   ├── type             类型：book / paper / article / original
│   ├── title            来源标题（如有）
│   └── location         具体位置（章节、页码等，可选）
├── raw_input            原始输入文本，永远保留，不做修改
├── processed            LLM 处理后的结构化内容
│   ├── proposition      核心命题（见下方说明）
│   └── open_questions   该想法留下的未解问题列表
├── tags                 内容性质标签（见下方说明）
├── links                与其他节点的连接列表
│   └── [{
│         target_id,     目标节点 ID
│         relation_type, 关系类型（支持 / 矛盾 / 延伸 / 来源 / 类比）
│         strength,      连接强度（LLM 评估，0.0 - 1.0）
│         confirmed      是否经用户确认（boolean）
│       }]
└── embedding            向量表示（用于语义检索，后期加入）
```

### 核心命题（Proposition）的自适应规则

核心命题是对原始输入的精炼，但其表述方式由输入内容的确定性决定：

| 输入特征 | 命题表述方式 | 示例 |
|----------|-------------|------|
| 明确断言 | 可被检验的陈述句 | "Agent 的自我模型可从记忆检索模式中涌现，无需独立模块" |
| 模糊联想 | 带不确定性的关系描述 | "Louise 的时间感知变化可能与 agent 持续时序感知后的认知结构变化存在结构上的相似性，机制尚不明确" |
| 问题记录 | 疑问形式保留 | "agent 的自我模型应该存在哪个层次？" |
| 纯粹碎片 | 最小化整理，接近原文 | 保留核心意象，注明"碎片，待发展" |

**原则**：命题的目的是提高未来检索时的精度，不是替代原始输入。如果精炼会导致失真，宁可保留模糊。

### 内容性质标签（Tags）

标签是**多选的、数据库性质的**，不构成层级结构，不决定存储位置。

| 标签 | 含义 |
|------|------|
| `definitive` | 有依据的明确结论或陈述 |
| `inferred` | 基于已有内容推断出的结论 |
| `vague` | 模糊感知，尚未能清晰表达 |
| `needs_thinking` | 明确需要进一步思考的问题 |
| `cross-domain` | 跨越多个主题领域的想法 |

## 三、数据流：从输入到存储

```
用户原始输入
      │
      ▼
 LLM 处理层
      ├── 1. 判断输入确定性 → 决定命题表述方式
      ├── 2. 生成核心命题（proposition）
      ├── 3. 提取开放问题（open_questions）
      ├── 4. 分配内容性质标签（tags）
      └── 5. 推断与已有节点的潜在连接（待用户确认）
      │
      ▼
  连接确认
      ├── 系统展示 LLM 推荐的连接（含关系类型和强度）
      ├── 用户确认、拒绝或修改
      └── 早期：用户主导确认；后期（节点成熟后）：可授权 LLM 自动决定
      │
      ▼
  双轨存储
      ├── Markdown 文件（Obsidian 可读，人类友好）
      └── 向量数据库 ChromaDB（语义检索用，后期加入）
```

## 四、查询模型

### 两段式查询语法

所有查询由两个部分组成：

```
[语义锚点]  +  [查询修饰符]
```

**语义锚点**：任意长度的文本，可以是关键词、一个句子、一段描述。  
系统以其语义含义进行检索，而不是关键字匹配。

**查询修饰符**（示例）：

| 修饰符 | 含义 |
|--------|------|
| `查找相关的全部内容` | 无时间和类型限制的全量语义检索 |
| `上周有什么和它有关` | 加时间范围过滤 |
| `我之前留下的相关问题` | 过滤 tag 为 `needs_thinking` 的节点 |
| `关于它的明确结论` | 过滤 tag 为 `definitive` 的节点 |
| `关于它的设计方案` | 过滤 relation_type 为"延伸/构想"的节点 |
| `和它矛盾的内容` | 过滤 relation_type 为"矛盾"的节点 |

### 查询执行逻辑

```
输入：[语义锚点] + [修饰符]
         │
         ▼
1. 将锚点向量化
2. 在向量数据库中检索语义最近的 N 个节点
3. 根据修饰符进行二次过滤（时间 / 标签 / 关系类型）
4. 按结果组织进入前端视图
```

## 五、前端视图设计

查询结果通过两种视图呈现，用户可切换。

### 5.1 聚合视图（Cluster View）

按内容性质标签将结果分组，每组为一个视觉"框"。

```
┌─ 明确结论 ─────────────────┐   ┌─ 开放问题 ──────────────────┐
│  · 节点 A（来源：论文X）   │   │  · 节点 D                   │
│  · 节点 B                 │   │  · 节点 E（2周前）           │
└────────────────────────────┘   └─────────────────────────────┘

┌─ 模糊猜想 ─────────────────┐   ┌─ 需要深入思考 ──────────────┐
│  · 节点 C                 │   │  · 节点 F                   │
└────────────────────────────┘   └─────────────────────────────┘
```

- 点击节点：展开查看完整内容（原始输入 + 命题 + 开放问题 + 连接列表）
- 框内支持按时间、连接数排序
- 支持从结果中选定某个节点作为新的语义锚点，发起扩展查询

**适合场景**：快速了解在某个主题下自己有哪些类型的积累。

---

### 5.2 认知地图视图（Epistemic Map）✅ 已实现

二维坐标系，同时呈现认知状态和主题分布。

**Y 轴（固定）**：认知确定性
```
确定
  │
  │   definitive
  │   inferred
  │   vague
  │   needs_thinking
  │
模糊
```

**X 轴（动态）**：由 LLM 聚类生成的主题区域，区域间距反映语义相关程度，随节点增长持续演化。

```
确定 │
     │  [AI 研究]           [哲学]              [科幻]
     │   · A   · B──· C      · G                 · K──· L
     │         └──· D        · H──· I
     │   · E                                · M
─────┼──────────────────────────────────────────────────→ 主题
     │         · F           · J
     │                                      · N──· O
模糊 │   · P                                    └──· Q
```

- 互相有 links 的节点聚成一个视觉团，孤立节点单独放置
- 主题区域标签由 LLM 自动生成，不需要用户预先定义
- 图整体随库的增长重新计算，是动态快照而非固定结构
- 点击节点行为与聚合视图相同
- 支持缩放、拖拽、节点点击展开详情

**适合场景**：全局观察自己思维的分布——哪些主题已经有明确积累，哪些停留在模糊阶段，哪些孤立想法尚未与任何东西连接。

**技术实现**: Cytoscape.js 力导向图
**访问方式**: 前端导航栏「认知地图」按钮

---

### 5.3 图形关系视图（Graph View）✅ 已实现

以网络图形式展示节点间的连接关系，突出知识网络的结构。

**可视化元素**:
- **节点圆点**: 代表知识节点，颜色按标签区分
- **连线**: 代表节点间的连接关系，粗细表示连接强度
- **关系标签**: 连线上的文字说明（支持/矛盾/延伸/来源/类比）
- **孤立节点**: 无连线的节点以灰色显示

**交互功能**:
- 拖拽移动节点
- 滚轮缩放画布
- 点击节点查看详情
- 双击节点聚焦其直接连接
- 筛选特定关系类型

**技术实现**: Cytoscape.js 力导向布局
**访问方式**: 前端导航栏「关系图」按钮

---

### 5.4 大纲视图（Outline View）✅ 已实现

独立于查询存在，是整个知识库的时间线鸟瞰。

```
[按时间线]

[2026-04] ── 节点 X ──→ 节点 Y ──→ 节点 Z
[2026-03] ── 节点 A
             └──→ 节点 B ──→ 节点 C（跨域连接）

[标签筛选]  [definitive] [needs_thinking] [cross-domain]
[排序]      按时间 / 按连接数 / 按主题
```

主要用途：
- 发现**孤立节点**（没有任何连接，可能是被遗忘的想法）
- 发现**高连接节点**（自己思维中真正的核心概念）
- 回顾某段时间内的思考轨迹

**访问方式**: 前端导航栏「大纲」按钮

## 六、技术栈

### 核心组件

| 层次 | 技术选择 | 理由 |
|------|---------|------|
| 输入 | 命令行 CLI | 最低摩擦，无需 GUI，桌面端快速录入 |
| 处理 | Python + MiniMax API | 已有基础，国内可用，推理在云端不占本地内存 |
| 后端 | FastAPI | 为 PWA 提供 API 接口，支持跨设备访问 |
| 存储（文件） | Markdown + Obsidian 目录 | 人类可读，版本控制友好 |
| 存储（向量） | ChromaDB（本地） | 轻量，无需服务器，阶段二加入 |
| 前端 | PWA（Progressive Web App） | 手机可添加到主屏幕，体验接近原生 App，无需上架应用商店，电脑浏览器同一地址访问 |
| 外网穿透 | Cloudflare Tunnel | 手机外网访问小主机，无需公网 IP，无需路由器配置，免费 |

### 部署结构

```
手机 / 电脑
    │
    │  HTTPS（Cloudflare Tunnel）
    ▼
Cloudflare 边缘节点
    │
    │  加密隧道
    ▼
小主机（Linux，7×24h）
    ├── FastAPI 后端（处理查询、调用 MiniMax API）
    ├── PWA 静态文件服务
    ├── ChromaDB（向量存储）
    └── Markdown 文件目录
```

### 内存占用预估

| 组件 | 预估占用 |
|------|---------|
| Linux 系统 | ~400MB |
| FastAPI 服务 | ~100MB |
| ChromaDB | ~200-500MB（随节点数增长） |
| Cloudflare Tunnel | ~50MB |
| **合计** | **~750MB - 1GB** |

4GB 限制内可行，有充足余量。LLM 推理（MiniMax）在云端运行，不计入本地内存。

### 明确排除的组件

| 组件 | 排除原因 |
|------|---------|
| Whisper | 输入瓶颈不在速度 |
| Quartz / 静态站点 | 无公开发布需求，PWA 已满足访问需求 |
| 原生 App | PWA 已能满足手机体验，无需开发和上架 |

## 七、实现阶段规划

系统按三个阶段递进实现，每个阶段有明确的完成标准。

### 阶段一：管道验证

**目标**：跑通最小化流程，验证"核心命题"格式是否真正适合使用习惯。

- 输入一段文字 → LLM 处理 → 生成节点结构 → 存储为 Markdown
- 连接推荐以文本形式呈现，用户手动确认
- 检索暂时使用 Obsidian 全文搜索
- 部署环境：小主机本地运行，仅局域网访问
- **完成标准**：积累 20-30 个节点，主观感受格式是否有价值

### 阶段二：语义检索 + 跨设备访问

**前提**：50 个节点以上。

- 所有节点向量化存入 ChromaDB
- 实现两段式查询接口，由 FastAPI 提供后端 API
- 部署 Cloudflare Tunnel，实现手机和外网访问
- 前端做成 PWA，手机可添加到主屏幕
- 查询结果以聚合视图呈现
- **完成标准**：能在手机上通过语义查询找到自己已经忘记存在的相关节点

### 阶段三：图形化视图与认知地图 ✅ 已完成

**前提**：100 个节点以上，查询已成为常规习惯。

- ✅ 实现聚合视图的完整图形化呈现
- ✅ 实现认知地图视图（动态坐标系聚类）
- ✅ 实现图形关系视图（网络图展示）
- ✅ 实现大纲视图（时间线鸟瞰）
- ⚪ 考虑 LLM 自动确认连接的授权机制（待开发）
- **完成标准**：能通过认知地图视图发现跨域连接

### 已实现但未在原规划中的功能

#### 主题提取与聚合系统
- 自动从节点内容中提取主题
- 多对一聚合：多个节点共享同一主题
- 主题按标签分类展示

#### 流式搜索进度条
- 实时显示搜索进度百分比
- 明确的阶段指示（提交→加载→LLM响应→处理→完成）
- 解决长耗时搜索（10-20秒）的用户体验问题

#### 多窗口节点详情系统
- 支持同时打开多个节点详情窗口
- 窗口层叠展示，方便对比
- 从连接直接打开相关节点

#### 批量处理队列可视化
- 批量添加节点时的处理状态显示
- 实时进度条和完成状态
- 错误处理和重试机制

#### 来源类型管理系统
- 支持自定义来源类型
- 内置类型：原创想法、书籍、论文、文章
- 可添加、编辑、删除自定义来源

## 八、关键设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 节点连接由谁决定 | LLM 推荐 + 用户确认，后期可授权 LLM 自动决定 | 早期需要把控质量，成熟后降低维护成本 |
| X 轴主题分类来源 | LLM 动态聚类生成 | 用户内容跨域，预先定义分类会限制系统适应性 |
| 原始输入是否保留 | 永远保留 | 命题精炼可能失真，原始输入是真实记录 |
| 向量检索时机 | 阶段二加入（50+ 节点后） | 节点量小时全文搜索已够用，过早引入复杂度无收益 |
| 手机访问方案 | PWA | 无需上架应用商店，体验接近原生，手机电脑同一套前端 |
| 外网穿透方案 | Cloudflare Tunnel | 小主机无公网 IP，Tunnel 是最低配置成本的解法 |
| 语音输入 | 不做 | 输入瓶颈不在速度 |
| 公开发布 | 不做 | 无此需求，PWA 仅供私人访问 |
| 界面语言 | 中文优先 | 用户为中文使用者，中文界面更符合思维习惯 |

## 九、界面语言规范

### 9.1 基本原则

**除必要的代码标识符、API 参数、命令名外，所有用户界面文本必须使用中文。**

这包括：
- CLI 输出信息（提示、状态、错误信息）
- 帮助文本和说明
- 标签和分类的显示名称
- 链接关系类型的显示名称

### 9.2 关系类型中英文对照

| 英文（代码/存储） | 中文（界面显示） |
|------------------|------------------|
| support | 支持 |
| contradict | 矛盾 |
| extend | 延伸 |
| source | 来源 |
| analogy | 类比 |

### 9.3 保留英文的场景

以下情况保持英文：
- CLI 命令名称（如 `add`, `list`, `show`）
- 选项参数名（如 `--type`, `--title`）
- 代码中的变量名、函数名
- 存储在文件中的原始数据（tags、relation_type 等）
- API 接口路径和参数

### 9.4 示例

**CLI 界面示例：**

```
╭───────────────────────────────── 知识库统计 ─────────────────────────────────╮
│ 节点总数: 5                                                                  │
│ 链接总数: 0                                                                  │
│ 已确认链接: 0                                                                │
│ 待处理链接: 0                                                                │
╰──────────────────────────────────────────────────────────────────────────────╯

标签分布:
  needs_thinking       ████ 4
  vague                ███ 3

潜在链接（待确认）:
  1. [类比] 人类工作记忆容量有限导致...
     理由: 两个概念在结构上有相似性...
  2. [支持] AI身份与责任划分...
     理由: 提供了补充论证...

请输入选择: 1,2
✓ 已确认 2 个链接
```
