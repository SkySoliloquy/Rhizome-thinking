# 修复 LLM 搜索功能计划

## 问题描述
用户在手机上测试时发现：
1. 进度条显示 10% 后约 1 秒直接跳到 100%
2. 看起来没有调用 LLM 进行语义重排序
3. 搜索 "ai" 只能找到唯一结果，而不是通过 LLM 语义理解的多个相关结果
4. 相关度调为 0 时返回全部节点（说明向量搜索在工作，但 LLM 重排序没工作）
5. 无法打开来源笔记

## 核心问题
**LLM 语义重排序功能没有被调用**，这是项目的核心功能。

## 诊断步骤

### 步骤 1: 检查 LLM 配置和环境变量
- 检查 `.env` 文件是否存在且包含 `minimax_api_key`
- 检查 `settings.minimax_api_key` 是否正确加载
- 确认 API Key 是否有效

### 步骤 2: 检查搜索优化器初始化
- 检查 `search_optimizer.py` 中的 `OptimizedSearch` 类初始化
- 确认 `LLMSearchReranker` 是否正确初始化
- 检查初始化失败时的回退逻辑

### 步骤 3: 检查搜索流程中的 LLM 调用点
- 检查 `search_with_timeout` 方法中的 LLM 调用逻辑
- 确认 `use_llm_rerank` 参数是否正确传递
- 检查 LLM 调用被跳过或失败的原因

### 步骤 4: 添加日志和调试信息
- 在关键位置添加日志输出，追踪 LLM 调用流程
- 确认请求是否发送到 MiniMax API
- 检查 API 响应和错误处理

### 步骤 5: 修复 LLM 调用问题
- 修复初始化问题
- 修复调用逻辑问题
- 确保进度条正确反映 LLM 处理阶段

### 步骤 6: 修复来源笔记打开问题
- 检查节点详情 API
- 检查前端打开来源笔记的逻辑

## 实施计划

1. **诊断阶段** - 检查配置和日志
2. **修复阶段** - 修复 LLM 调用逻辑
3. **验证阶段** - 测试搜索功能
4. **优化阶段** - 改进错误处理和用户体验

## 预期结果
- 进度条应该在 25% 左右停留较长时间（等待 LLM 响应）
- 搜索结果应该基于 LLM 语义理解而非仅向量相似度
- 相关度阈值应该有效过滤结果
- 可以正常打开来源笔记查看详情
